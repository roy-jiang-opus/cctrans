## ✨ विशेषताएँ

- 🪞 **इनलाइन द्विभाषी प्रदर्शन** — अनुवाद जवाब की स्ट्रीमिंग के साथ हर अंग्रेज़ी पंक्ति के नीचे दिखता है
- 🧩 **तीन लेआउट** — पंक्ति-दर-पंक्ति अंतर्संयोजन, ब्लॉक-वार (`cctrans mode section`), या पूरा जवाब एक साथ (`cctrans mode message`)
- 🔄 **नीचे जोड़ें या जगह पर बदलें** — अनुवाद को अंग्रेज़ी के नीचे दिखाएँ, या `cctrans display replace` से सिर्फ़ अनुवाद उसकी जगह पर दिखाएँ
- ❓ **अनुवादित प्रश्न संवाद** — Claude Code के इंटरैक्टिव प्रश्न-संवाद भी आपकी भाषा में दिखते हैं, जबकि मॉडल फिर भी आपका अंग्रेज़ी उत्तर ही पढ़ता है
- 🧾 **गैर-विनाशकारी** — ट्रांसक्रिप्ट और मॉडल कॉन्टेक्स्ट शुद्ध अंग्रेज़ी में रहते हैं; skills, दस्तावेज़, कोड अप्रभावित
- 🆓 **मुख्य लूप के शून्य टोकन** — अनुवाद एक अलग सस्ते (मुफ़्त विकल्प सहित) बैकएंड से होता है, Claude Code सत्र से पूरी तरह बाहर
- ⌨️ **इनपुट अनुवाद (beta)** — अपनी भाषा में लिखें, मॉडल अंग्रेज़ी में काम करता है — और अंग्रेज़ी में जवाब देता है (`cctrans input on`)
- 🌏 **10 लक्ष्य भाषाएँ** — `zh-Hans` `zh-Hant` `ja` `ko` `ru` `hi` `es` `pt` `fr` `de`
- 🔌 **6 बैकएंड + स्वतः fallback** — OpenAI / Anthropic / DeepL / Azure / मुफ़्त Google / आपकी अपनी Claude सदस्यता
- 📁 **प्रति-प्रोजेक्ट ओवरराइड** — किसी रिपॉज़िटरी में `.cc-translate.json` केवल उसी प्रोजेक्ट के लिए भाषा/मोड बदल देता है (या अनुवाद बंद कर देता है)
- 🔒 **कुंजी अलगाव** — API कुंजियाँ केवल chmod-600 फ़ाइल में; शेल पर्यावरण चर कभी नहीं पढ़े जाते
- 🛟 **फ़ेल-सेफ़** — कोई भी त्रुटि या टाइमआउट शुद्ध अंग्रेज़ी पर वापस आ जाता है; सत्र कभी नहीं रुकता
- 🩺 **बिल्ट-इन डायग्नोस्टिक्स** — `cctrans doctor` बताता है कि कुछ अनुवादित क्यों नहीं हो रहा; `cctrans stats` दिखाता है कि आपने कितने टोकन बचाए
- 🎚️ **इंटरैक्टिव सेटिंग्स** — `cctrans settings` एक ही स्क्रीन का संपादक खोलता है (एरो कुंजियाँ); स्पेसिंग, मार्कर, मॉडल और बहुत कुछ समायोजित करें, बेसिक + एडवांस्ड

## 🚀 त्वरित शुरुआत

```bash
npm install -g cctrans@latest && cctrans install
```

इंस्टॉल हुक पंजीकृत करता है और एक इंटरैक्टिव सेटिंग्स संपादक खोलता है (भाषा, प्रदर्शन मोड, बैकएंड, API कुंजी — `cctrans settings` इसे कभी भी फिर से खोल देता है)। फिर **Claude Code पुनः आरंभ करें** — जवाब द्विभाषी हो जाएँगे। Claude Code के इनपुट बॉक्स में `!cctrans off` / `!cctrans on` टाइप करके कभी भी टॉगल करें (`!` CC का बिल्ट-इन bash मोड है — कोई मॉडल कॉल नहीं, कोई टोकन नहीं)।

**पहले से इंस्टॉल है?** `npm install -g cctrans@latest` से अपडेट करें — यह अगले जवाब से ही लागू हो जाता है (हुक हर चंक पर डिस्क से ताज़ा चलता है); आपकी सेटिंग्स, कुंजियाँ और पंजीकृत हुक अछूते रहते हैं, दोबारा सेटअप की ज़रूरत नहीं।

## 🤔 यह प्रोजेक्ट क्यों

दो समस्याएँ, एक आर्किटेक्चर:

**1. Claude Code बार-बार अंग्रेज़ी में जवाब देता है।** Skills और दस्तावेज़ अंग्रेज़ी में ही रहने चाहिए, और CLAUDE.md में "हिन्दी में जवाब दो" लिखने पर भी जवाब अंग्रेज़ी में लौट आते हैं। दोबारा अनुवाद माँगना एक पूरा मॉडल टर्न खर्च करता है और बातचीत का इतिहास भी दूषित करता है।

**2. अपनी भाषा में काम करने पर छिपा हुआ टोकन-कर लगता है — ख़ासकर Claude पर।** वही अर्थ व्यक्त करने में अंग्रेज़ी से **~1.5–3× ज़्यादा टोकन** लगते हैं (Claude का टोकनाइज़र गैर-लैटिन लिपियों को खराब संपीड़ित करता है), और Claude Code की 5-घंटे की विंडो और साप्ताहिक सीमाएँ टोकन में मापी जाती हैं — यानी गैर-अंग्रेज़ी सत्र आपका प्लान 1.5–3× तेज़ी से जलाते हैं। अहम बात: **जवाब की गुणवत्ता समस्या नहीं है**: Claude बहुभाषी बेंचमार्क में >90% स्कोर करता है। दर्द विशुद्ध रूप से लागत का है।

| | जापानी | कोरियाई | रूसी | हिन्दी | चीनी |
|---|---|---|---|---|---|
| अंग्रेज़ी की तुलना में टोकन लागत | ~2–3× | ~2–3×+ | ~1.5× | ~2–3×+ | ~2–3× |

भाषा-समायोजित सीमाओं के लिए Anthropic का issue ([#26401](https://github.com/anthropics/claude-code/issues/26401)) *not planned* कहकर बंद कर दिया गया — कोई आधिकारिक उपाय नहीं है।

**इसलिए सबसे सस्ता और सही डिज़ाइन ठीक वही है जो यह टूल करता है:** सत्र शुरू से अंत तक अंग्रेज़ी में रहता है (प्रॉम्प्ट, ट्रांसक्रिप्ट, मॉडल कॉन्टेक्स्ट — मुख्य लूप का शून्य अतिरिक्त टोकन), और आपकी भाषा केवल वहीं मौजूद है जहाँ इंसान को पढ़ना है: हर अंग्रेज़ी पंक्ति के नीचे केवल-प्रदर्शन अनुवाद पंक्ति, जिसे एक अलग सस्ता बैकएंड रेंडर करता है।

स्रोतों सहित पूर्ण शोध नोट्स: [MOTIVATION.md](MOTIVATION.md)।

## ⚙️ यह कैसे काम करता है

Claude Code के नेटिव **`MessageDisplay` हुक** (v2.1.152+) पर आधारित: यह हर असिस्टेंट संदेश के रेंडर होते समय सक्रिय होता है और पूर्ण टेक्स्ट खंड (`delta`) हुक को सौंपता है; हुक द्वारा लौटाया गया `displayContent` **केवल स्क्रीन प्रदर्शन को बदलता है**, संग्रहीत संदेश को कभी नहीं।

```
Claude अंग्रेज़ी स्ट्रीम करता है
        │  हर पूर्ण खंड पर सक्रिय (stdin: turn_id/message_id/index/final/delta)
        ▼
  hook/message-display.js  ──►  src/interleave.js  ──►  src/translate.js
   (delta पढ़ें, टॉगल जाँचें)    (गद्य/कोड/पहले से लक्ष्य भाषा)   (बैकएंड + कैश)
        │
        ▼  displayContent = "अंग्रेज़ी पंक्ति\n↳ अनुवाद पंक्ति" लौटाता है
   Claude Code प्रदर्शन को उसी जगह बदल देता है (मूल ट्रांसक्रिप्ट/कॉन्टेक्स्ट में रहता है)
```

> CC 2.1.169 पर सत्यापित: `delta` खंड **गैर-अतिव्यापी** पूर्ण खंड हैं (संचित टेक्स्ट नहीं), सामान्य `\n` दोनों भाषाओं को अलग पंक्तियों में दिखाता है, और कोड ब्लॉक / पथ / पहले से लक्ष्य भाषा वाली पंक्तियाँ स्वतः छोड़ दी जाती हैं।

## 🧩 प्रदर्शन मोड

`line` (डिफ़ॉल्ट) अंतर्संयोजन करता है: हर अंग्रेज़ी पंक्ति के नीचे अनुवाद की पंक्ति, जवाब की स्ट्रीमिंग के साथ। `section` अंग्रेज़ी को ठीक वैसे ही रखता है जैसे Claude स्ट्रीम करता है और **ब्लॉक पूरा होने पर एक समूहीकृत अनुवाद** जोड़ देता है — सूची-भारी जवाबों के लिए कहीं ज़्यादा शांत। `message` एक क़दम और आगे जाता है: पूरा जवाब शुद्ध अंग्रेज़ी में स्ट्रीम होता है और **एक समूहीकृत अनुवाद बिल्कुल अंत में आता है**:

```
Use these flags:
↳ ये फ़्लैग इस्तेमाल करें:

- Enable the cache
- Set a small timeout
- Prefer the batch API
  ↳ कैश सक्षम करें
  ↳ छोटा टाइमआउट सेट करें
  ↳ batch API को प्राथमिकता दें
```

```bash
cctrans mode section   # प्रति ब्लॉक · cctrans mode message — पूरा जवाब · cctrans mode line — डिफ़ॉल्ट पर वापस
```

> section/message मोड में अनुवाद **ब्लॉक (या पूरा जवाब) पूरा होने पर** दिखता है, स्ट्रीमिंग के दौरान नहीं — धीमे बैकएंड (जैसे `claude-code`, 3–6s/कॉल) के साथ यह ठहराव महसूस होता है, इसलिए यहाँ API बैकएंड सबसे अच्छे रहते हैं। अगर किसी ब्लॉक का अनुवाद विफल हो जाए, तो अंग्रेज़ी अप्रभावित रहती है और वह ब्लॉक बस बिना अनुवाद के रह जाता है।

**नीचे जोड़ें या जगह पर बदलें।** डिफ़ॉल्ट रूप से अनुवाद अंग्रेज़ी के *नीचे* दिखाया जाता है (द्विभाषी)। सिर्फ़ अपनी भाषा में पढ़ना चाहते हैं? `cctrans display replace` हर अंग्रेज़ी पंक्ति की **जगह पर** अनुवाद दिखाता है:

```bash
cctrans display replace   # सिर्फ़ अनुवाद · cctrans display append — द्विभाषी पर वापस
```

Replace **line मोड** में काम करता है (section/message अपनी बनावट के कारण पहले अंग्रेज़ी स्ट्रीम करते हैं, इसलिए बदलने को कुछ नहीं होता)। दोनों ही स्थितियों में ट्रांसक्रिप्ट और मॉडल का संदर्भ 100% अंग्रेज़ी में रहते हैं; जिस पंक्ति का अनुवाद न हो सके, वह अपना मूल पाठ बनाए रखती है, इसलिए कभी कुछ ग़ायब नहीं होता।

## ❓ प्रश्न संवाद

जब Claude Code आपसे विकल्पों में से चुनने को कहता है (इंटरैक्टिव प्रश्न-संवाद), तो प्रश्न, विकल्पों के लेबल और उनके विवरण भी आपकी भाषा में दिखाए जाते हैं — और आपका चुना हुआ उत्तर फिर भी मॉडल तक **अंग्रेज़ी** में ही पहुँचता है, इसलिए उसका तर्क केवल अंग्रेज़ी में बना रहता है:

```
 ☐ 颜色偏好
Which color do you prefer?
↳ 您更喜欢哪种颜色？
❯ 1. Red
     ↳ 红色
     A bold, vibrant color
     ↳ 大胆、鲜艳的颜色
   2. Blue
     ↳ 蓝色
```

यह Claude Code के `PreToolUse`/`PostToolUse` हुक पर सवार होता है (प्रश्न-संवाद टूल इनपुट से रेंडर होता है, जहाँ तक संदेश ओवरले नहीं पहुँच सकता)। यह आपकी `display` सेटिंग का पालन करता है — append मोड में द्विभाषी, replace मोड में केवल आपकी भाषा। डिफ़ॉल्ट रूप से चालू; इसे `cctrans dialog off` से बंद करें। अगर कोई संवाद समय पर अनुवादित न हो सके, तो वह बिना बदलाव के अंग्रेज़ी में दिखता है।

> **अपग्रेड कर रहे हैं?** अपडेट करने के बाद नए संवाद हुक रजिस्टर करने के लिए एक बार `cctrans install` चलाएँ। उत्तर-पुनर्स्थापन (मॉडल का उत्तर अंग्रेज़ी में रखना) के लिए Claude Code ≥ 2.1.121 चाहिए — `cctrans doctor` पुराने संस्करणों पर चेतावनी देता है।

| `anthropic` | `cctrans key anthropic` | ~1s/खंड | उच्च | `claude-haiku-4-5` + structured outputs, सख़्त समान-लंबाई पंक्ति सरणियाँ (~$0.0005/खंड) |

| `azure` | `cctrans key azure` (मुफ़्त: 20 लाख वर्ण/माह) | ~0.5s/खंड | मध्यम-उच्च | वैकल्पिक `cctrans key azure-region` |

| `claude-code` | `claude` CLI लॉग-इन | ~3-6s/खंड | उच्च | आपकी **Claude सदस्यता** पर चलता है (`claude -p` headless) — कोई अतिरिक्त लागत नहीं पर धीमा |

प्राथमिक बैकएंड विफल/टाइमआउट होने पर शृंखला **स्वतः google पर fallback** करती है — सत्र कभी अवरुद्ध नहीं होता। हर अनुवादित पंक्ति बैकएंड + भाषा + सामग्री हैश से कैश होती है।

API कुंजियाँ **केवल** `~/.cc-translate/keys.json` (chmod 600) में रहती हैं — उन्हें `cctrans setup` / `cctrans key` से सेट करें या फ़ाइल को सीधे संपादित करें। शेल के पर्यावरण चर कभी नहीं पढ़े जाते, इसलिए इस टूल की कुंजियाँ और टर्मिनल की कुंजियाँ एक-दूसरे को दूषित नहीं करतीं।

बाक़ी सेटिंग्स (बैकएंड, भाषा, मार्कर, मॉडल, Azure एंडपॉइंट) `~/.cc-translate/state.json` में हैं — `cctrans` कमांड से बदलें या फ़ाइल को सीधे संपादित करें।

## 🗣 भाषाएँ

```bash
cctrans lang zh-Hans  # सरलीकृत चीनी (डिफ़ॉल्ट)    cctrans lang zh-Hant  # पारंपरिक चीनी
cctrans lang ja       # जापानी                     cctrans lang ko       # कोरियाई
cctrans lang ru       # रूसी                       cctrans lang hi       # हिन्दी
cctrans lang es       # स्पेनिश                     cctrans lang pt       # पुर्तगाली
cctrans lang fr       # फ़्रेंच                      cctrans lang de       # जर्मन
```

**CJK + रूसी + हिन्दी** (गैर-लैटिन लिपियों) के लिए "यह पंक्ति पहले से लक्ष्य भाषा में है" Unicode रेंज से मुफ़्त में पहचानकर छोड़ दी जाती है। **स्पेनिश / पुर्तगाली / फ़्रेंच / जर्मन** (लैटिन लिपियों) के लिए पहचान इसके बजाय एक संयमी stopword ह्यूरिस्टिक से होती है — और अगर पहले से लक्ष्य भाषा वाली कोई पंक्ति दोबारा अनुवाद के लिए चली भी जाए, तो identity जाँच उस दोहराव को दबा देती है, यानी बदतरीन स्थिति में बस एक व्यर्थ बैकएंड कॉल, ग़लत पंक्ति कभी नहीं। ध्यान दें कि लैटिन-लिपि भाषाओं के लिए टोकन बचत कम है (अंग्रेज़ी की तुलना में ~1.1–1.2×, जबकि गैर-लैटिन के लिए 1.5–3× — देखें [MOTIVATION.md](MOTIVATION.md)); उनके लिए असली आकर्षण द्विभाषी प्रदर्शन ही है।

चीनी के लिए BCP-47 **लिपि कोड** (`zh-Hans`/`zh-Hant`) इस्तेमाल होते हैं — पारंपरिक चीनी एक लिपि है, क्षेत्र नहीं; `zh-CN` / `zh-TW` उपनाम के रूप में स्वीकार्य हैं और स्वतः सामान्यीकृत होते हैं। भाषा बदलना तुरंत प्रभावी होता है (हुक हर कॉल पर स्थिति पढ़ता है); हर भाषा का कैश स्वतंत्र है।

## 📁 प्रति-प्रोजेक्ट ओवरराइड

किसी रिपॉज़िटरी की रूट में `.cc-translate.json` रखें (वर्किंग डायरेक्टरी का कोई भी पैरेंट चलेगा) ताकि वैश्विक सेटिंग्स केवल उसी प्रोजेक्ट के लिए ओवरराइड हों:

```json
{ "target": "ja", "mode": "section" }
```

या किसी ख़ास प्रोजेक्ट के लिए `{ "enabled": false }` से ओवरले बंद कर दें। ओवरराइड-योग्य फ़ील्ड: `enabled`, `target`, `mode`, `backend`, `marker`, `model`, `inputEn`, `inputMinChars`। सीक्रेट ओवरराइड नहीं होते — कुंजियाँ `~/.cc-translate/keys.json` में ही रहती हैं, और एंडपॉइंट सेटिंग्स जान-बूझकर केवल वैश्विक स्तर पर ही रखी गई हैं। `cctrans status` (प्रोजेक्ट के अंदर चलाने पर) और `cctrans doctor` दोनों दिखाते हैं कि प्रोजेक्ट ओवरराइड कब सक्रिय है। किसी क्लोन की गई रिपॉज़िटरी की `.cc-translate.json` को उसके कोड का हिस्सा मानें: उदाहरण के लिए, यह उस रिपॉज़िटरी में होने वाले काम के लिए बैकएंड बदल सकती है (जिसमें `claude-code` भी शामिल है, जो आपकी सदस्यता खर्च करता है)।

## 🩺 समस्या-निवारण

ओवरले डिज़ाइन से ही फ़ेल-सेफ़ है: हर त्रुटि सत्र को रोकने के बजाय शुद्ध अंग्रेज़ी पर उतर आती है — जिसका मतलब यह भी है कि विफलताएँ **चुपचाप** होती हैं। जब कुछ भी अनुवादित न हो रहा हो:

```bash
cctrans doctor
```

यह जाँचता है: हुक पंजीकरण (पुराने इंस्टॉल के बासी पथ सहित), आपका Claude Code संस्करण (MessageDisplay के लिए ≥ 2.1.152 चाहिए), कॉन्फ़िगर किया गया बैकएंड और उसकी कुंजी, लाइव कनेक्टिविटी (लेटेंसी सहित), और **हुक की आख़िरी त्रुटि** (स्ट्रीम के बीच कुछ विफल होने पर हुक उसे `~/.cc-translate/last-error.json` में दर्ज करते हैं)। और यह देखने के लिए कि ओवरले ने आपके लिए अब तक क्या किया है:

```bash
cctrans stats    # अनुवादित पंक्तियाँ + मुख्य लूप के बचाए गए टोकन का अनुमान
cctrans cache    # अनुवाद-कैश का आकार; प्रबंधन के लिए clear / gc (डिफ़ॉल्ट सीमा 200 MB)
```

## ⌨️ इनपुट अनुवाद (beta)

`cctrans input on` एक `UserPromptSubmit` हुक सक्षम करता है: जब प्रॉम्प्ट में पर्याप्त गैर-लैटिन वर्ण हों (डिफ़ॉल्ट 4+ — पूर्ण गिनती, इसलिए फ़ाइल पथ और आइडेंटिफ़ायर ट्रिगर को कमज़ोर नहीं करते; `cctrans input threshold <n>` से समायोजित करें), तो अंग्रेज़ी अनुवाद संदर्भ के रूप में मॉडल को संलग्न होता है जिसे वह आधिकारिक निर्देश मानता है, और मॉडल से **अंग्रेज़ी में जवाब देने** को कहा जाता है — इससे द्विभाषी ओवरले काम करता रहता है और बातचीत का संदर्भ अंत तक अंग्रेज़ी में रहता है। (CC 2.1.169 पर सत्यापित: हुक प्रॉम्प्ट को फिर से लिख नहीं सकते, इसलिए मूल इतिहास में रहता है और अंग्रेज़ी साथ में।) अंग्रेज़ी इनपुट अपरिवर्तित गुज़रता है; किसी त्रुटि पर प्रॉम्प्ट सुरक्षित रूप से ज्यों का त्यों भेजा जाता है।

> **Beta**: अनुवाद कॉल हर गैर-अंग्रेज़ी प्रॉम्प्ट के सबमिशन को ~0.5–1.5 सेकंड रोकती है। डिफ़ॉल्ट रूप से बंद; setup विज़ार्ड एक बार पूछता है। प्रतिक्रिया → [issues](https://github.com/roy-jiang-opus/cctrans/issues)

## 📏 व्यवहार और सीमाएँ (सत्यापित)

- हुक **स्ट्रीमिंग के दौरान** हर खंड पर सक्रिय होता है; हर खंड का अनुवाद होकर उसी जगह बदला जाता है — अनुवाद अंग्रेज़ी के साथ-साथ क्रमिक रूप से दिखता है।
- हुक का टाइमआउट **10 सेकंड** है; यह टूल आंतरिक रूप से 9s पर गार्ड करता है। कोई भी त्रुटि / टाइमआउट / अधिक लंबाई (>9,000 वर्ण) **सुरक्षित रूप से मूल अंग्रेज़ी पर fallback** करती है — सत्र कभी नहीं अटकता।
- हर अनुवादित पंक्ति सामग्री हैश से **कैश** होती है (`~/.cc-translate/cache`, 200 MB की सीमा रोज़ाना लागू होती है); री-पेंट और दोहराए टेक्स्ट की लागत शून्य। सभी मोड एक ही कैश साझा करते हैं।
- section/message मोड में चल रहे (अभी अधूरे) ब्लॉक का टेक्स्ट `~/.cc-translate/msgstate` में बफ़र होता है (डिस्क पर वही एक्सपोज़र जो कैश का है); संदेश पूरा होने पर फ़ाइल हटा दी जाती है और पुरानी फ़ाइलें 24 घंटे बाद साफ़ कर दी जाती हैं।
- `openai` के साथ हर खंड लगभग एक API कॉल (~$0.0001) और शुद्ध अंग्रेज़ी की तुलना में ~1s/खंड की देरी; `google` तेज़ है पर गुणवत्ता थोड़ी कम।
- **Markdown तालिकाएँ** बरकरार रहती हैं: तालिका बिना बदले ज्यों-की-त्यों गुज़रती है (जिससे Claude Code की नेटिव बॉक्स-ड्रॉइंग रेंडरिंग सुरक्षित रहती है) और ठीक उसके बाद तालिका की एक अनुवादित प्रति दिखाई जाती है — अब बीच में डाले गए अनुवाद से पंक्तियाँ टूटती नहीं।
